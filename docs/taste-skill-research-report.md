> 基于 `Leonxlnx/taste-skill` v2 SKILL.md（1206 行）的深度分析
> v3 — 完整翻译 Taste Skill 全部 14 节 + 附录，补全 v2 遗漏的 §0.C/D、§2 设计系统映射、§3 默认架构、§6 性能/无障碍、§7 Dial 定义、§8 暗色协议、§13 适用范围

## 1. Taste Skill 核心机制

Taste Skill 是一个面向前端开发的 Agent Skill，解决的核心问题是：**LLM 在没有约束时总是走最省力的设计路径，产出千篇一律的模板化界面。**

它通过四层机制解决这个问题：

### 1.1 Brief Inference（需求推断）

**机制**：强制 LLM 在写任何代码前，先输出一行 Design Read 声明。

```
"Reading this as: B2B SaaS landing for technical buyers,
 with a Linear-style minimalist language, leaning toward
 Tailwind utilities + Geist + restrained motion."
```

**读取 6 个信号**：
1. 页面类型（landing / portfolio / redesign / editorial）
2. 氛围词（"minimalist", "Awwwards", "brutalist"）
3. 参考信号（URL、截图、竞品名）
4. 受众（B2B 采购 vs 设计消费者 vs 招聘官）
5. 已有品牌资产（logo、色、字体、摄影）
6. 隐性约束（无障碍优先、公共部门、受监管行业）

**本质**：不让 LLM 跳到默认方案，强制它先"读题"。

**0.C 模糊时只问一个问题，不要猜**：
- 当 Design Read 真正存在分歧时，只问一个问题，永远不要一次性抛多个问题
- 示例：*"这个更偏 Linear 简洁还是 Awwwards 实验性？"*
- 如果能从上下文自信推断，**不要问**，直接声明 Design Read 继续

**0.D 反默认纪律**：
- 不要默认使用：AI-purple 渐变、暗色网格上的居中 Hero、三等分 feature 卡片、通用 glassmorphism、无限循环微动画、Inter + slate-900
- 这些是 LLM 默认值。根据 Design Read 刻意越过它们

### 1.2 Three Dials（三旋钮）

**机制**：把模糊的"风格"量化为三个维度，每个 1-10。

| 旋钮 | 含义 | 1 | 10 |
|------|------|---|----|
| DESIGN_VARIANCE | 布局实验性 | 完美对称 | 艺术混乱 |
| MOTION_INTENSITY | 动效深度 | 静态 | 电影级物理 |
| VISUAL_DENSITY | 信息密度 | 画廊留白 | 驾驶舱密集 |

默认值 `8/6/4`，但根据 Design Read 自动调整：

| 信号 | VARIANCE | MOTION | DENSITY |
|------|----------|--------|---------|
| "minimalist / calm / Linear-style" | 5-6 | 3-4 | 2-3 |
| "playful / Awwwards / experimental" | 9-10 | 8-10 | 3-4 |
| "trust-first / public-sector" | 3-4 | 2-3 | 4-5 |

**本质**：量化设计意图，防止 LLM 用模糊的"风格"掩盖偷懒。

### 1.3 AI Tells（AI 痕迹黑名单）

**机制**：列出 50+ 种 LLM 设计的统计性特征，全部硬性禁止。

**颜色类**：禁止 AI-purple、禁止高端消费 beige+brass+espresso、强调色最多 1 个
**字体类**：禁止 Inter 默认、禁止 Fraunces/Instrument_Serif、衬线限制使用
**布局类**：禁止居中 Hero、禁止三等分卡片、Eyebrow 每 3 section 最多 1 个
**内容类**：禁止填充动词、假精确数字、模板品牌名
**装饰类**：完全禁止 em-dash（—）、禁止假截图、禁止装饰性状态点

**本质**：这些不是品味问题，是从真实生产测试中总结的 LLM 统计性特征。

### 1.4 Pre-Flight Check（起飞前检查）

60+ 项机械检查清单，输出代码前必须逐项通过。任何一项不通过就不算完成。

## 2. Taste Skill 的底层原理

### 2.1 约束即质量

> **品味的提升不是"加"上去的，是"减"出来的。堵住所有省力的默认路径后，LLM 被迫探索，输出自然落在更好的区间。**

Taste Skill 的 research/ 目录有实证支撑：
- LLM 的截断/偷懒是行为选择，不是能力不足（Experiment B 证实）
- 上下文退化不是主要原因（200 轮对话测试中模型保持了关键指令）
- 懒惰的根因是 RLHF 训练中的简洁偏好 + 经济约束

### 2.2 制造摩擦

四层机制的本质都是在**制造摩擦**：

| 机制 | 摩擦类型 | 防止什么 |
|------|----------|----------|
| Brief Inference | 认知摩擦 — 强制先思考再动手 | 跳到默认方案 |
| Three Dials | 量化摩擦 — 强制声明具体值 | 用模糊风格掩盖偷懒 |
| AI Tells | 否定摩擦 — 堵住 50+ 条捷径 | 走最省力路径 |
| Pre-Flight Check | 验证摩擦 — 60+ 项机械检查 | 偷工减料 |

### 2.3 生成性 vs 约束性

Taste Skill 的规则是**约束性的**（告诉你不能做什么），不是**生成性的**（告诉你该做什么）。但约束本身有生成效果 — 当所有省力路径被堵住，LLM 被迫在更好的空间中搜索。

## 3. PPT Skill 的现状分析

### 3.1 SKILL.md 内容构成

当前 SKILL.md（634 行）的内容分布：

| 类别 | 行数 | 占比 |
|------|------|------|
| 技术参考（API、CLI、格式、参数） | ~400 | 63% |
| 执行流程（5 步工作流） | ~60 | 10% |
| 设计指导（Content Design Rules） | ~15 | 2% |
| 系统能力描述（atoms、diagrams、images） | ~100 | 16% |
| 约束/注意事项 | ~60 | 9% |

**设计指导仅占 2%。** Taste Skill 的设计指导占 90%。

### 3.2 当前设计指导的局限

Content Design Rules 的 8 条规则全是**功能性的** — 告诉 LLM "做什么会触发什么渲染效果"，但没有告诉 LLM **"为什么这样做"和"什么时候不该这样做"**。

### 3.3 LLM 在管线中的角色

| 步骤 | LLM 做什么 | 设计决策参与度 |
|------|-----------|---------------|
| Step 1 | 理解需求、规划页面结构 | **高** — 决定叙事弧、页面数、每页 goal |
| Step 2 | 生成 3 个风格提案 | **中** — 选择 style 参数 |
| Step 3 | 写 content.json | **高** — 决定标题、要点、卡片内容 |
| Step 4 | 调用管线生成 | **低** — 传参执行 |

**LLM 的设计影响力集中在 Step 1 和 Step 3** — 叙事规划和内容创作。

### 3.4 ui-ux-pro-max 的角色

ui-ux-pro-max 提供的是安全级约束（保证不犯错），不是卓越级约束（保证不平庸）。当 LLM 脱离约束自由发挥时，因为缺少卓越级约束，作品质量反而下降。

## 4. PPT 版设计工程指令（完整翻译 Taste Skill §4）

以下将 Taste Skill §4 的每一条设计指令翻译为 PPT 版本。这些指令应写入 SKILL.md，约束 LLM 在 Step 1/3 的决策。

### 4.1 排版纪律（Typography → PPT Typography）

**字号层级**：
- 封面标题：44-52pt，最多 2 行
- 内页标题：32-36pt
- 副标题/正文：14-16pt
- 要点/列表：13-14pt
- 注释/图注：11-12pt
- **禁止**：全 PPT 只有 2 种字号（标题+正文），必须有至少 4 级字号层级

**字体选择纪律**：
- **禁止** Calibri/Arial 作为默认字体 — 和 Taste Skill 禁止 Inter 同理，这是 PPT 的 AI 默认字体
- **优先**：使用系统自带的优质字体（Windows: Segoe UI / Georgia / Consolas; Mac: San Francisco / Avenir / Menlo）
- **衬线体限制**：只在 editorial / luxury / heritage / publication 场景使用。不在 tech / startup / data-heavy 场景使用衬线体
- **"创意=衬线"是 AI 默认思维** — 创意提案可以用 Geist/Cabinet Grotesk 等无衬线展示体，不一定要衬线
- **同一 PPT 内禁止混用超过 2 种字体族**（标题字体 + 正文字体，最多加一个等宽字体用于代码）

**标题强调规则**：
- 要强调某个词时，用**加粗或变色**（同一字体族内），不要混用衬线+无衬线做"对比强调"
- 混字体强调是业余表现 — 如标题用无衬线，突然一个词换成衬线

**行距与降部**：
- 斜体标题中含降部字母（y g j p q）时，行距不能太紧，否则字母尾巴被裁切
- PPT 中行距至少 1.1 倍

### 4.2 配色校准（Color Calibration → PPT Color）

**强调色规则**：
- 全 PPT 最多 **1 个强调色**，饱和度适中（不过分鲜艳）
- 强调色用于：标题下渐变线、featured 卡片顶部条、badge、progress bar
- **一致性锁**：一旦选定强调色，全 PPT 每一页都必须使用同一强调色。第 7 页突然换色是断裂

**PPT 的"AI-Purple"等价物**：
- **默认蓝色渐变封面** — PPT 中最常见的 AI 默认，等同于前端的 AI-purple
- **禁止**：未指定风格时自动选择蓝色系。如果用户说"做一个专业 PPT"，不应该默认蓝色
- **覆盖路径**：如果品牌本身是蓝色的（IBM、Facebook），蓝色当然可以用

**高端消费品配色禁令**（PPT 版 Taste Skill "Premium-Consumer Palette Ban"）：
- LLM 对"高端/奢侈品/精致"类 PPT 的默认配色：深色背景 + 金色强调 + 白色文字
- 具体禁止的色值族：
  - 背景：`#1A1A2E`, `#16213E`, `#0F3460`（深蓝夜色系）
  - 强调：`#C9A96E`, `#D4AF37`, `#B8860B`（金色/古铜色系）
  - 文字：`#FFFFFF` on `#1A1A2E`（深底白字万能组合）
- **替代方案（轮换，不要连续用同一家族）**：
  - **冷调奢华**：石墨灰 + 银白 + 冰蓝强调
  - **森林**：深绿 + 骨白 + 琥珀强调
  - **黑与暖棕**：纯黑 + 暖棕 + 锐利对比
  - **钴蓝 + 奶白**：饱和蓝 + 单一中性色
  - **纯单色 + 一个饱和跳色**：灰白 + 灰黑 + 一个亮色（翠绿/电蓝/玫红）
- **轮换规则**：如果上一次高端 PPT 用了金色系，这次必须换一个家族

**配色一致性锁**：
- 一个 PPT 内冷暖灰不混用 — 要么全冷灰（Slate/Zinc），要么全暖灰（Stone/Warm Gray）
- 数据图表中的颜色必须从 PPT 主色板派生，不要用彩虹色

**投影对比度规则**（PPT 特有，Web 不需要）：
- 暗色主题中，文字与背景对比度要比 Web 更高 — 投影仪会吃掉对比度
- 前景色在暗底上至少比背景亮 60%（亮度差），Web 的 WCAG AA 不够
- 浅色主题中，避免浅灰文字 on 白色背景 — 投影仪上消失

### 4.3 布局多样化（Layout Diversification → PPT Layout）

**反居中偏见**：
- VARIANCE > 4 时，避免所有内容页都居中标题+居中要点
- 使用：左对齐标题 + 左对齐内容 / sidebar-left 布局 / asymmetric 布局
- **覆盖**：section divider 和 keynote 风格的宣言页可以居中

**反等权卡片**：
- features 页 3 张卡片不允许完全等权 — 第一张必须 featured（更大标题、顶部渐变条）
- 4+ 张卡片时考虑分组：2 大 + 2 小，或 1 hero + 3 标准

**Hero（封面/尾页）纪律**：
- **Hero 必须适配初始视口**：标题最多 2 行，副标题最多 20 字，CTA 文字可见
- **Hero 元素上限**：最多 4 个文本元素 — (1) 小标签 or 品牌条（选一个，不要两个都有），(2) 主标题，(3) 副标题，(4) CTA
- **Hero 中禁止**：在 CTA 下面加一行小字标签、信任标志条、定价提示、要点列表
- **封面标题字号纪律**：标题 6 个词以上不要用 52pt — 先降字号或缩短标题，4 行封面标题永远是字号错误
- **Hero 顶部间距上限**：标题离顶部最多 pt-24 当量，更多的话内容浮在页面中间看起来像 bug

**"Used by / 客户"logo 墙**：
- 属于封面之后的第一页，不属于封面内部
- 封面是价值主张 + 主 CTA 的空间，logo 墙是独立的信任页

### 4.4 材质、阴影、卡片（Materiality → PPT Shapes）

**卡片使用条件**：
- 只在需要表达层级关系时使用卡片 — 否则用分割线或留白分组
- DENSITY > 7 时，避免通用卡片容器 — 数据指标用纯排版更透气

**阴影规则**：
- 使用阴影时，阴影色相应该偏背景色 — 不要在浅色背景上用纯黑投影
- 暗色主题用发光（glow）代替阴影

**圆角一致性锁**：
- 全 PPT 选择一种圆角体系并坚持：
  - 全直角（radius 0）— 工业/技术/brutalist
  - 全圆角（8-12pt）— 友好/现代/通用
  - 全胶囊（pill）— 活泼/消费/创意
- 混合体系只在有文档化规则时允许（如"按钮用 pill，卡片用 12pt，输入框用 6pt"），且规则必须全程遵守
- 圆角按钮在直角布局中，或直角卡片在 pill 按钮页中，都是设计断裂

### 4.5 交互状态（Interactive UI States → PPT Page States）

PPT 没有交互状态，但有**页面角色**。LLM 默认只写"内容页"，不规划页面角色。

**必须规划的页面角色**：
- **呼吸页**：信息量低，视觉留白多。用于高密度页之后，给观众消化时间
- **过渡页（section divider）**：话题切换的视觉信号。大号数字 + 标题 + 渐变线
- **数据冲击页**：一个巨大数字 + 一行注释。用于核心数据点
- **视觉锚点页**：全出血图片为主，文字极少。用于情感转折
- **总结/行动页（cta）**：清晰下一步。不能只是"谢谢"

**角色交替规则**：
- 连续 3 页不能都是"标题+要点"的内容页
- 高密度数据页后必须有低密度页
- 6+ 页 PPT 至少 1 个 section divider

### 4.6 数据与列表模式（Data & Form Patterns → PPT Data Pages）

**长列表规则**：
- 5 个以下要点：单列列表
- 6+ 个要点：触发双栏布局（已有功能）
- 10+ 个要点：不要用列表 — 用卡片网格 / 表格 / 组件库中的 infographic
- **禁止**：20+ 行的数据表在 PPT 中展示 — PPT 是演示不是文档

**规格表（Spec Sheet）的 PPT 版**：
- LLM 对"产品规格"页的默认：标题 + 10 行要点，每行一个规格
- **禁止**此模式。替代方案：
  - **2 列卡片网格**：每个规格一张卡片，规格名+大号数值+一句"为什么重要"
  - **重点+折叠**：3-4 个核心规格大号展示，其余收在"更多规格"下
  - **分组**：10 个规格分 3 组，每组一个标题和一条分隔线

### 4.7 布局纪律（Layout Discipline → PPT Slide Discipline）

**Section-Layout 重复禁令**：
- 一旦使用了某种布局族（如 3 列卡片、全出血图+文字、左右分屏），该族在 PPT 中最多出现 1 次
- "我们的产品"和"核心优势"不应该看起来一样
- 8 页 PPT 至少使用 4 种不同的视觉布局

**左右交替上限**：
- 连续"左图右文 → 左文右图"交替最多 2 次
- 第 3 次连续交替是 Pre-Flight Fail — 必须用全宽、垂直堆叠、bento 网格或其他布局打破

**Eyebrow 限制**（PPT 版 Taste Skill "Eyebrow Restraint"）：
- "Eyebrow"在 PPT 中是标题上方的小号大写标签（如 `核心优势`、`03 / 产品介绍`）
- **每 3 页最多 1 个 eyebrow**
- 如果第 1 页有 eyebrow，第 2、3 页不能有
- **代替 eyebrow 的做法**：直接去掉。标题本身够了。如果需要分类，页面在 PPT 中的位置已经分类了
- **Section 编号 eyebrow 禁止**：`00 / 目录`、`001 · 核心能力`、`06 · 工作方式` — 这些是 AI 痕迹，禁止

**Split-Header 禁令**：
- "左边大标题 + 右边小段解释文字"的页面头部模式，**禁止作为默认**
- 如果确实需要标题+解释，垂直堆叠（标题在上，正文在下，最大 65 字宽）
- 只有右侧承载视觉/交互元素时才用 split-header

**Bento 背景多样性**：
- Bento/网格布局不能 6 张全白卡片只放文字
- 至少 2-3 个格子有真实视觉变化：图片、品牌渐变、纹理背景、着色底
- 白底白字 bento 只有排版没有视觉 = AI 默认

### 4.8 图片与视觉资产策略（Image Strategy → PPT Visual Assets）

**PPT 是视觉产品。纯文字页面不是极简，是不完整。**

**图片优先级**：
1. **AI 生成图片**：用 fetch_image() 为每个关键页面生成专属图片。不要跳过因为"写代码更快"
2. **素材库图片**：用户提供 images/ 目录中的图片
3. **最后手段**：告诉用户"这个 PPT 在以下位置需要真实图片"

**极简也需要图**：
- 即使是极简/线性风格，至少 2-3 页需要真实图片（封面、一个产品/场景图、一张支撑图）
- 可以生成黑白极简摄影，不要因为 dial 低就跳过所有图片

**假产品截图禁止**：
- 不要用文本框拼出假的 dashboard / 终端 / 应用界面 — 这是 PPT 中最大的 AI Tell
- 替代：用真实截图、AI 生成的产品图、或完全跳过产品预览用文字描述

**封面必须有视觉**：
- 纯文字 + 渐变色块 = 占位符，不是封面
- 至少需要：真实图片、AI 生成图、或强有力的排版设计（大字+留白本身就是视觉）

**品牌 logo 墙**：
- 使用真实 logo 图片，不要用文字替代（`<span>腾讯</span>` 排一排）
- Logo 墙只放 logo，不放行业/类别标签（不要在 `华为` 下面加 `通信`）

### 4.9 内容密度（Content Density → PPT Content Density）

**PPT 靠第一印象，不靠完整阅读。刀刀见血。**

**默认内容形态**：
- 每页：短标题（≤ 8 词）+ 短副文（≤ 25 字）+ 一个视觉资产 OR 一个 CTA
- 超过这个形态的，必须有理由

**不堆砌数据**：
- 20 行出版表、30 行获奖列表、巨型定价矩阵在 PPT 中 = 错误布局
- 替代：Top 3-5 亮点 + "完整列表见附录"

**文案自审（Copy Self-Audit）**：
- 交付前重读每个可见字符串（标题、副标题、badge、正文、图注、页脚）
- 标记以下问题的字符串并重写：
  - **语法不通**："以AI之力赋能未来"
  - **指代不清**："我们将持续保持" — 保持什么？
  - **AI 幻觉**：强行隐喻、不成立的类比
  - **LLM 式深沉**：故作谦逊、伪工匠标签、假装诗意的微元数据
- **如果不确定一个字符串是否合理，换成平实的功能句。AI 生成的巧妙文案比无聊文案更糟。**

**假精确数字**：
- `92%`、`4.1×`、`48k`、`5.8mm` — 要么来自真实数据，要么标记为示例，要么禁止
- 不要伪造品牌没有声称的工程精度

**单一语感**：
- 不要在同一 PPT 中混用技术体（`47 tasks · 0.6 ctx-switches/day`）、散文体、营销体
- 除非品牌调性明确要求混搭

### 4.10 引用与证言（Quotes → PPT Testimonials）

- 引文主体最多 **3 行**，永远不要 6 行
- 如果原始引文更长 → 截取。PPT 引用是片段，不是完整评论
- 署名：姓名 + 职位 +（可选）公司。永远不要只有姓名
- 引号：用真正的排版引号（""），不要用直引号（""）

### 4.11 主题锁定（Theme Lock → PPT Theme Consistency）

**一个 PPT 一个主题，页面之间不反转。**

- 如果是暗色主题，所有页面都是暗色。不要在暗色页面中间插入一个亮色页面
- 如果是亮色主题，同理
- **例外**：如果用户明确要求"色彩块故事"或"滚动切换主题"，允许做一次主题切换 — 但必须是有意图的强过渡，不是随机交替
- 同一主题家族内的背景色微调可以（如 `#0A1E3D` 和 `#0F2847`），但不要从 `#0A1E3D` 跳到 `#FFF8F0`

## 5. PPT 版动效纪律（翻译 Taste Skill §5）

### 5.1 动效必须有动机

在给任何页面添加动效前，问自己："这段动效传达什么？"
- **有效回答**：层级（引导注意力到关键内容）、叙事（按序列揭示内容匹配叙事）、反馈（确认用户动作）
- **无效回答**："看起来酷"

### 5.2 声称的动效必须可见

如果 motion > 4，PPT 中必须真的有动效 — 至少封面入场、关键页面入场。
一个声称 `motion: 7` 但完全静态的 PPT 是断裂的。
反之，如果无法交付工作动效，把 motion 降到 2-3，交付干净的静态 PPT。

### 5.3 Marquee 上限

PPT 中滚动文字条（用动效模拟 marquee）最多 1 页 1 个。两个以上 marquee = 偷懒填充。

### 5.4 减少动画偏好

如果 audience 可能包含对动效敏感的人群（公共部门、无障碍优先场景），motion 默认降到 2-3。

## 6. PPT 版 AI Tells 黑名单（翻译 Taste Skill §9.F）

以下模式来自真实 LLM 生成 PPT 的测试。它们是模型试图"看起来有设计感"时的默认签名。**除非用户明确要求，否则全部硬性禁止。**

### 6.1 封面与首页

- **禁止封面版本标签**：`V0.6`、`v2.0`、`BETA`、`内测版`、`抢先体验` — 只在产品发布/预览场景允许
- **禁止"品牌 · No.01"式副标签**：如 `Marrow · No.01 · 6-quart` 式微元数据行 — 直接去掉

### 6.2 Section 编号与微标签

- **禁止 section 编号 eyebrow**：`00 / 目录`、`001 · 核心能力`、`06 · 工作方式` — eyebrow 用自然语言命名主题，不要编号
- **禁止图片/卡片上的编号标签**：`01 / 04`、`1/3` — 观众会数数，不需要标签
- **禁止"Index of Work, 2018-2026"式范围标签**作为 eyebrow — 直接说这个 section 是什么

### 6.3 分隔符与点

- **中间点（·）限量**：每行元数据中最多 1 个。不要把 `·` 当默认分隔符（`foo · bar · baz · qux`）
- **禁止装饰性彩色状态点**：每个导航项/列表行/badge 前面的彩色圆点 — 默认禁止。只在传达真实语义状态时允许（如"在线"/"离线"指示）

### 6.4 Em-Dash 禁令（最高违规率）

**Em-dash（—）在 PPT 中完全禁止。** 这是 LLM 最明显的签名。

- 标题中禁止 — 用句号或逗号
- Eyebrow / badge / 正文 / 署名 / 图注中禁止 — 用普通连字符（-）或换行
- 中文场景中同样禁止 — 用破折号（——）也是 AI 痕迹，用逗号或分号代替
- 日期范围用连字符：`2018-2026`，不用 em-dash

### 6.5 排版花招

- **禁止换行+斜体标题**作为"设计手法"：如 "for thirty\n*years.*" 式拆分 — 标题应该自然阅读，不要为了"设计感"硬拆
- **禁止竖排旋转文字**：`核心竞争力` 旋转 90° — agency 式陈词滥调
- **禁止十字准线/细网格线作为装饰**：为了"显得有设计感"画的纵横线 — 禁止。只在组织真实内容时使用

### 6.6 假产品预览

- **禁止假产品 UI**：用文本框拼的假 dashboard、假终端、假任务列表 — 这是 PPT 中 #1 AI Tell
- **禁止假版本页脚**：`v0.6.2-rc.1`、`last sync 4s ago` 在假截图中 — 除了是 AI 签名之外什么都不是

### 6.7 营销文案 Tell

- **禁止"静默使用中"/"默默服务"**式社会证明标题 — 用自然语言："客户包括"/"被以下机构采用"，或跳过标题让 logo 说话
- **禁止"来自一线"/"实战笔记"/"当前在研"式伪工匠标签**在引言/博客/sidebar 区 — 用平实功能标签（"客户评价"、"最新动态"）或跳过标签
- **禁止天气/城市/时间条**：`北京 14:23 · 18°C` — 除非场景确实是全球分布式团队或旅行品牌
- **禁止 eyebrow 下的微元数据句**：如 *"以下是我们今天交付的能力，而非路线图承诺。列表将保持精简。"* 这种塞在 section 标题下的句子是杂乱
- **禁止通用步骤标签**："阶段 1 / 阶段 2 / 阶段 3"、"步骤 01 / 步骤 02 / 步骤 03" — 禁止。用动词+名词直接命名（"安装"、"配置"、"交付"），不要"阶段 1：安装"

### 6.8 标签/版本戳

- **禁止图片上叠加标签**：不要在照片上覆盖 `品牌 · 02`、`场景 - 演示` 等 span 标签 — 让图片自己说话，或把说明文字放在图片下方
- **禁止装饰性图片署名**：`实地考察 No.12 · Ines Caetano`、`场景 III · 35mm` 式的伪纪实署名 — 除非有真实摄影师署名，否则跳过或用一行功能说明
- **禁止营销页上的版本页脚**：`v1.4.2`、`Build 0048` — 这是开发者工具信息，不是演示文稿内容
- **禁止"已预约 412/800"式库存计数器**作为装饰 — 只在有限量候补的正式场景使用

### 6.9 装饰文字条

- **禁止封面底部的装饰文字条**：`品牌. 创新. 技术.`、`设计 / 创造 / 交付`、`始建于2018 · 北京 · 品牌. 创新. 技术.` 这种小号大写字母条 — agency 式陈词滥调
- **禁止 section 标题右上角漂浮的小段解释文字**：左边一个大标题，右上角对不齐地浮着一段小字 — 这是 Tell。要么把解释文字直接放在标题下方，要么做干净的 2 列头部

### 6.10 列表、分隔线与评分

- **禁止长列表每行都有分隔线**：10 行规格表每行下面一条线是最偷懒的布局 — 见 §4.6 的替代方案
- **禁止带填充背景轨道的评分/进度条**作为对比视觉：需要展示"X / Y"对比时，用一个数字+小图标，或微型内联条（无背景轨道）

### 6.11 地域/时间/滚动提示

- **城市名/时间/天气条在 99% 的场景中禁止**
- **滚动提示禁止**：`Scroll`、`↓ 向下滚动`、`滚动探索` — PPT 中更不需要，观众知道怎么翻页
- **装饰性状态点零容忍**

## 7. PPT 版设计词汇表（翻译 Taste Skill §10）

这些是 LLM 应该**知道名字**的设计模式。命名一个模式 = LLM 在决策时可以"调用"它，而不是凭直觉。当前 LLM 只知道 8 种 goal type，词汇量太少。

### 7.1 封面范式

| 模式名 | 描述 | 适用场景 |
|--------|------|----------|
| **Asymmetric Split Hero** | 文字一侧，视觉资产另一侧，大量留白 | SaaS、agency、高端消费品 |
| **Editorial Manifesto Hero** | 大字排版，无图片，近乎海报 | 宣言式发布、品牌主张 |
| **Full-Bleed Image Hero** | 全出血图 + 渐变叠加 + 文字 | 有高质量图片的场景 |
| **Kinetic-Type Hero** | 动态排版作为主视觉 | 创意 agency、Awwwards 风 |
| **Data-Impact Hero** | 一个巨大数字 + 一行注释 | 数据驱动的 pitch、traction 页 |
| **Minimal Typography Hero** | 纯文字，极致留白，字体即设计 | 极简/Linear 风格 |

### 7.2 内页布局

| 模式名 | 描述 | 适用场景 |
|--------|------|----------|
| **Sidebar + Content** | 左侧窄侧边栏（品牌色）+ 右侧内容 | 咨询报告、McKinsey 风 |
| **Split Text-Image** | 左文右图或左图右文 | 产品介绍、场景描述 |
| **Bento Grid** | 不等大网格，混合内容类型 | 功能展示、特性概览 |
| **Big Number Focus** | 一个巨大数字居中 + 注释 | 核心数据点、KPI |
| **Card Row** | 横排卡片，首张 featured | features 展示 |
| **Comparison Split** | 左右对比 | before/after、竞品对比 |
| **Timeline Horizontal** | 水平时间线 + 里程碑 | 路线图、发展历程 |
| **Quote Spotlight** | 大字引言 + 署名 | 客户评价、名人引言 |
| **Code Terminal** | 深色代码块 + 语言 badge | 技术演示、开发者受众 |
| **Full-Width Visual** | 全出血图/渐变 + 少量文字 | 情感转折、视觉锚点 |

### 7.3 数据呈现

| 模式名 | 描述 | 适用场景 |
|--------|------|----------|
| **Table Diagram** | 交替行色表格 | 结构化数据对比 |
| **Chart Focus** | 图表为主 + 标题 | 趋势、分布、对比 |
| **Metric Dashboard** | 多个小指标卡片 | KPI 总览、运营数据 |
| **Infographic Component** | 组件库信息图 | 流程、层级、SWOT |
| **Number Grid** | 2×2 或 2×3 大数字网格 | 多个关键指标 |

### 7.4 内容关系 → 视觉策略映射

这是 Taste Skill 没有但 PPT 特别需要的 — **从内容语义推理视觉形式**：

| 内容关系 | 视觉策略 | 示例 |
|----------|----------|------|
| 递进/因果 | Timeline 或 numbered steps | 问题→方案→成果 |
| 对比/二分 | Comparison Split | 传统 vs 创新 |
| 主次/层级 | 不等权布局（1 大 + N 小） | 核心优势 + 3 个辅助优势 |
| 等权并列 | Card Row 或 Grid | 4 个等价特性 |
| 包含/层级 | Hierarchy 组件或 Pyramid | 组织架构、概念层级 |
| 证据支撑 | 中心论点 + 环绕支撑 | 核心主张 + 3 个证据 |
| 流程/循环 | Cycle 或 Process 组件 | 开发流程、运营闭环 |
| 数据驱动 | Big Number Focus 或 Chart | 关键指标、市场数据 |

**LLM 在 Step 3 写 content.json 时应该先判断要点间的关系，再决定 goal type 和 component_category，而不是只看内容主题选 goal。**

## 8. PPT 版重设计协议（翻译 Taste Skill §11）

### 8.1 模式检测

- **Greenfield** — 全新 PPT，从 Dial 基线出发
- **Redesign - Preserve** — 保留品牌，视觉现代化。先审计，提取品牌 token，逐步演进
- **Redesign - Overhaul** — 新视觉语言，保留内容。视觉上等同于 greenfield

如果不确定，问一次："这次重设计是保留现有风格，还是视觉上从头开始？"

### 8.2 审计优先（beautify 模式适用）

在修改任何内容之前，记录当前状态：
- **品牌 token** — 主色/强调色、字体、logo 位置、圆角
- **信息架构** — 页面顺序、section 划分、核心转化路径
- **内容块** — 哪些在做功，哪些是填充
- **保留的模式** — 标志性布局、可识别的视觉签名
- **淘汰的模式** — AI slop tell、断裂布局、低质量图、通用文案
- **当前 Dial 值推断** — 从现有 PPT 推断 variance/motion/density，作为起点而非基线

### 8.3 保留规则

- 不要改变信息架构（除非被要求）
- 提取品牌色后再应用配色规则 — 已经是蓝色的品牌保持蓝色
- 保留文案语感（除非被要求重写）
- 尊重现有无障碍成果 — 不要退化对比度、可读性

### 8.4 现代化杠杆（优先级排序）

按顺序应用 — 满足需求即停：
1. **字体刷新** — 最大视觉提升 / 最低风险
2. **间距与节奏** — 增加 section padding，修复垂直节奏
3. **配色校准** — 降饱和、统一中性色、保留品牌强调色
4. **动效层** — 增加 motion-appropriate 的入场动画
5. **封面与关键页重构** — 用 §7 的词汇表重构
6. **全页替换** — 只在现有页不可救药时

### 8.5 永不静默更改

未经用户明确批准，不修改：
- 页面顺序
- 核心导航标签
- 品牌 logo 或 wordmark
- 现有法律/合规文案

## 9. PPT 版 Pre-Flight Check（翻译 Taste Skill §14）

生成 PPT 前必须逐项检查。**任何一项不通过，PPT 不算完成。**

### 基础质量
- [ ] 所有字体 ≥ 11pt
- [ ] 所有页面 ≥ 3 个 shape
- [ ] 所有页面有文本内容
- [ ] 图片正确放置（hero→hook, product→features）
- [ ] 无损坏的图片链接

### 设计一致性
- [ ] 强调色贯穿全 PPT（不中途换色）
- [ ] 圆角体系一致（全 sharp / 全 soft / 全 pill，不混用）
- [ ] 字体对一致（标题字体 + 正文字体，全篇统一）
- [ ] 暗色/亮色主题不中途切换
- [ ] 冷暖灰不混用（全冷 or 全暖）
- [ ] 字号层级 ≥ 4 级（不是只有标题+正文两种）

### 排版纪律
- [ ] 封面标题 ≤ 2 行
- [ ] 封面副标题 ≤ 20 字
- [ ] 内页标题 ≤ 2 行（超长标题降字号或缩写）
- [ ] 无换行+斜体"设计手法"
- [ ] 无竖排旋转文字
- [ ] 斜体标题有降部清理（行距 ≥ 1.1）

### 布局多样性
- [ ] 相邻 2 页不同布局族
- [ ] 8+ 页 PPT 至少使用 4 种不同视觉布局
- [ ] features 页卡片不等权（第一张 featured）
- [ ] 不存在连续 3 页相同密度的要点
- [ ] 无连续 3 次"标题+要点"同构页面
- [ ] 左右交替 ≤ 2 次

### Eyebrow 与标签
- [ ] Eyebrow 数量 ≤ ceil(页面数 / 3)
- [ ] 无 section 编号 eyebrow（`00 / INDEX`）
- [ ] 无图片上叠加标签
- [ ] 无装饰性状态点
- [ ] 无通用步骤标签（"阶段 1/2/3"）

### 配色
- [ ] 非指定蓝色场景中，封面不是默认蓝色渐变
- [ ] 高端场景配色不是默认金+深蓝
- [ ] 数据图表颜色从主色板派生（非彩虹色）
- [ ] 暗色主题中文字对比度足够（投影级）
- [ ] 浅色主题中无浅灰字 on 白底

### 叙事节奏
- [ ] 有至少 1 个 section divider（6+ 页时）
- [ ] hook 和 cta 视觉上有区分
- [ ] 信息密度有变化（不是每页 3 要点）
- [ ] 高密度页后有低密度页（呼吸节奏）
- [ ] 每页核心信息 ≤ 1 个（不堆砌）

### 内容质量
- [ ] 无填充动词（赋能/领先/一站式/生态/全方位/革新/引领）
- [ ] 无假精确数字（除非有数据来源或标记为例示）
- [ ] 要点之间有逻辑关系（不是随机罗列）
- [ ] 无 em-dash（—）或中文破折号（——）作为设计元素
- [ ] 无装饰文字条（`品牌. 创新. 技术.`）
- [ ] 无假产品截图
- [ ] 引文 ≤ 3 行
- [ ] 引文有完整署名（姓名+职位）

### 视觉资产
- [ ] 封面有真实视觉（不是纯文字+渐变色块）
- [ ] 极简风格也有 ≥ 2-3 页有图片
- [ ] Bento/网格中 ≥ 2-3 格有视觉变化（不是全白文字卡片）
- [ ] Logo 墙用真实图片，不用文字替代

### 场景适配
- [ ] 投影场景：文字对比度足够
- [ ] 打印场景：关键信息不依赖动效
- [ ] 大屏场景：关键数字 ≥ 36pt

### 动效
- [ ] 每段动效有动机（能用一句话解释为什么存在）
- [ ] motion > 4 时 PPT 中真的有动效
- [ ] Marquee ≤ 1 个/页

### Dial 值合理性
- [ ] Dial 值从 Design Read 推理得出，不是静默使用基线
- [ ] variance > 4 的 PPT 中确实有不对称布局
- [ ] density 值在页面间有变化（不是所有页面同密度）

### 避免范围外场景
- [ ] 不是纯数据表格（应该用 Excel）
- [ ] 不是多步表单/向导（应该用应用 UI）
- [ ] 不是实时协作界面

## 9.1 PPT 版设计系统映射（翻译 Taste Skill §2）

Taste Skill §2 规定了什么时候用官方设计系统（Fluent/Carbon/Material 等），什么时候用原生 CSS。PPT 的等价物是：**什么时候用组件库（component_library），什么时候用 PrecisionRenderer 内置布局**。

| 场景 | 应该用 | 为什么 |
|------|--------|--------|
| 咨询/金融报告 | sidebar 布局 + component_library process/hierarchy | 专用图表比手绘专业 |
| 技术分享 | 代码块 + component_library infographic | 技术受众需要信息密度 |
| 教育/培训 | exercise 页 + 内置 bullets | 简单结构更适合教学 |
| 创意提案 | blocks 自定义布局 + AI 图片 | 需要视觉冲击，模板会限制 |
| 品牌发布 | 全出血图片 + minimal 文字 | 品牌表达靠视觉不靠文字 |

**一个 PPT 一次只用一种设计体系。** 不要在同一个 PPT 中混用 McKinsey sidebar 风格和 Cyberpunk neon 风格。

## 9.2 PPT 版默认架构与惯例（翻译 Taste Skill §3）

Taste Skill §3 规定了前端的默认技术栈（React + Tailwind + Motion + Fonts）。PPT 的等价物：

**默认管线**：StoryPlanner → DesignDecider → ContentGenerator → PrecisionRenderer

**字体策略**：
- 优先使用系统字体（确保跨平台可用）
- 不要链接 Google Fonts（PPT 中不可靠）
- 中英文必须有 CJK 伴侣字体映射（已有 theme_mapper.py）

**图标策略**：
- PPT 中用组件库的 SmartArt 图标，不要手绘 SVG
- 一个 PPT 内图标风格统一（同一线宽、同一填充风格）

**Emoji 策略**：
- 默认不使用 emoji 替代专业图标
- 只在用户明确要求活泼/社交风格时使用，且少量有意图地使用

**响应式（PPT 特化）**：
- PPT 16:9 是固定画布，不需要响应式布局
- 但需要考虑：投影仪裁切（安全边距 0.5"）、打印缩放（确保 11pt+ 字体可读）
- 宽屏 16:9 vs 标准 4:3 — 默认 16:9，4:3 需要显式指定

**依赖验证**：
- 生成 python-pptx 代码前，确认 package 已安装
- 引用组件库前，确认 component_library 存在
- 引用图片前，确认图片路径有效

## 9.3 PPT 版性能与无障碍护栏（翻译 Taste Skill §6）

### 硬件加速
- 动效只用于 transform 和 opacity — PPT 中这是自动处理的
- 避免每页超过 50 个 shape — 影响 PowerPoint 渲染性能

### 减少动画（PPT 版 `prefers-reduced-motion`）
- motion > 3 时必须考虑：受众是否包含对动效敏感的人群
- 公共部门/无障碍优先场景，motion 默认 2-3
- 如果不确定受众，motion 不要超过 5

### 暗色模式
- 消费者面向的 PPT 应该默认设计两种主题
- 但 PPT 不像 Web 有系统偏好切换 — 由用户在 Step 2 选定一种
- 选定后全 PPT 锁定该主题

### 文件大小与性能
- 单页 shape 数 < 50（过多 shape 导致 PowerPoint 卡顿）
- 图片预裁切（Pillow cover-fit），不要拉伸
- 图片缓存优先（同关键词不重复生成）
- 组件库组件比手绘更轻量（坐标归一化，复用模板）

### Z-Index 约束（PPT 版 layer 顺序）
- PPT 中 shape 的 z-order 决定覆盖关系
- 不要随意调整 z-order — 遵循系统性规则：背景层 < 内容层 < 装饰层 < 覆盖层

## 9.4 PPT 版 Dial 定义（翻译 Taste Skill §7）

Taste Skill §7 给出了三个 Dial 的精确行为定义。PPT 版：

### DESIGN_VARIANCE（1-10）

| 级别 | PPT 行为 |
|------|----------|
| 1-3 可预测 | 对称布局，等宽卡片，居中标题，等距 padding |
| 4-7 偏移 | 左对齐标题，不等权卡片，sidebar 布局，图片位置变化 |
| 8-10 不对称 | 不等大网格，大幅留白区域，侧边偏移内容，非标准比例 |

### MOTION_INTENSITY（1-10）

| 级别 | PPT 行为 |
|------|----------|
| 1-3 静态 | 无自动动效，只有 slide transition |
| 4-7 流畅 | fade-in 入场，stagger 出现，subtle 强调 |
| 8-10 编排 | fly-in 入场，grow CTA，按序列出现，强动效表达 |

### VISUAL_DENSITY（1-10）

| 级别 | PPT 行为 |
|------|----------|
| 1-3 画廊 | 大量留白，大间距（py-32 to py-48 当量），贵而干净 |
| 4-7 日常 | 标准间距（py-16 to py-24 当量） |
| 8-10 驾驶舱 | 紧凑 padding，无边框卡片，1px 线分隔，数字用等宽字体 |

**重要**：这些 Dial 值必须从 Design Read 推理得出，不是静默使用基线 8/6/4。在 SKILL.md 中声明 Dial 值时，必须同时声明推理理由。

## 9.5 PPT 版暗色模式协议（翻译 Taste Skill §8）

### 策略选择（选一个，坚持用）

| 策略 | 适用场景 |
|------|----------|
| **暗色优先** | 科技/赛博朋克/数据展示，默认 `background: #0A1E3D` |
| **亮色优先** | 商务/教育/咨询，默认 `background: #FFFFFF` |
| **用户选择** | 消费品/品牌发布，Step 2 让用户选 |

### 不指定具体色值

Taste Skill §8.B 明确说"不要在这里规定具体的暗色模式颜色"，因为品牌和需求决定。只强制：
- **对比度**：WCAG AA 最低线，投影场景要求更高
- **层级对等**：亮色中 CTA 突出，暗色中也必须突出
- **品牌忠实**：主色在暗色模式中保持可辨识，不要把品牌色降饱和到消失
- **无纯黑纯白**：不用 `#000000` 和 `#FFFFFF` — 用近黑和近白，纯值杀死深度

### 测试

交付前在两种模式下都检查一遍（如果是双主题项目）。不要只看过一种模式就交付。

## 9.6 PPT 版适用范围（翻译 Taste Skill §13）

**PPT Skill 不适用于**：

| 场景 | 应该用什么 |
|------|-----------|
| 纯数据表格/财务报表 | Excel |
| 多步表单/向导 UI | Web 应用 |
| 实时协作界面 | 专门的应用 |
| 交互式仪表盘 | Power BI / Tableau |
| 长文档（>50 页） | Word / PDF |

如果用户的需求属于以上场景，**明确告知**，指出正确工具，只对适合 PPT 的展示层面应用本 skill。

| 布局多样性 | 相邻页不同族 + 8页≥4种 | 量化约束 | **中等** — 依赖 LLM 词汇量 |

**中等确定性提升**（效果依赖 LLM 推理能力）：

| 改进项 | 不确定性来源 |
|--------|-------------|
| 叙事节奏感 | 需要 LLM 理解"呼吸页""数据冲击页"等概念并正确规划 |
| 内容关系驱动布局 | 需要 LLM 先判断关系类型再选 goal，多一步推理 |
| 文案质量 | AI Tells 能禁填充词，但"好文案"不是靠禁止产生的 |

**低确定性提升**（效果难以预测）：

| 改进项 | 不确定性来源 |
|--------|-------------|
| Dial 值对布局的实际影响 | Dial 是声明式的，PrecisionRenderer 是否读取并响应 Dial 值？当前代码可能不读取 |
| 动效质量 | 取决于管线动效能力，不是 SKILL.md 能解决的 |
| 跨会话配色轮换 | 无记忆机制 |

### 13.3 三个关键风险

#### 风险 1：SKILL.md 膨胀导致 LLM 遵守率下降

1370 行的 SKILL.md 对 LLM 是大上下文消耗。Taste Skill 原版 1206 行面向的是前端 Agent（通常有更大上下文），PPT Skill 的 LLM 调用可能更短促。

**缓解**：分层组织 — 核心规则（AI Tells + Pre-Flight + 一致性锁）放前面且精简，扩展规则（词汇表 + Dial 定义 + 重设计协议）放后面。LLM 不需要每次都读全文。

#### 风险 2：约束性规则可能让输出趋同

50+ 条 AI Tells + 55 项 Pre-Flight Check + 各种"禁止"，可能让 LLM 的搜索空间过窄，产出"安全但无聊"的结果。

**缓解**：对冲方法是 §7 设计词汇表 — 每禁止一条捷径，提供一条替代路径。当前词汇表 20+ 模式足够对冲，但需要验证 LLM 是否真的会"调用"这些模式名。如果 LLM 只遵守禁止但不使用替代路径，输出会更安全但不会更高级。

#### 风险 3：Dial 机制是声明式而非执行式（最关键）

Design Read + Three Dials 要求 LLM 声明值，但当前 PrecisionRenderer 的 `render_slide()` **不接受 Dial 参数**。Dial 值写了但代码不读，就是装饰。

**缓解**（二选一）：
- **方案 A**：在 PrecisionRenderer 中增加 Dial 响应逻辑（代码改动）— 让 variance 实际影响布局选择、density 实际影响间距、motion 实际影响动效选择
- **方案 B**：在 SKILL.md 中明确 Dial 只约束 LLM 的内容规划决策（不约束渲染），避免虚假承诺 — 例如 "variance > 4 时 LLM 应该在 content.json 中使用更多样的 goal type"，而不是 "variance > 4 时 renderer 会生成不对称布局"

**推荐方案 B**（先期）— 不改代码，让 Dial 约束 LLM 的规划行为而非渲染行为。方案 A 是后续代码改进。

### 13.4 修正后的总体评估

| 维度 | v3 原评估 | 审查后评估 | 修正原因 |
|------|-----------|-----------|----------|
| 理论正确性 | 9/10 | 9/10 | 不变 |
| 实施可行性 | 8/10 | **7/10** | Dial 执行链和动效能力是短板 |
| 预期效果 | 8/10 | **8/10** | AI Tells + 一致性锁立竿见影，叙事/节奏需迭代 |
| 副作用风险 | 6/10 | **6/10** | 输出僵化 + SKILL.md 膨胀是真实风险 |
| 投入产出比 | 9/10 | **9/10** | 只改 SKILL.md 不改代码，成本极低 |

### 13.5 建议执行顺序

| 阶段 | 内容 | 风险 | 收益 | 前置条件 |
|------|------|------|------|----------|
| **Phase 1** | AI Tells + Pre-Flight Check + 一致性锁 | 低 | 高 | 无 |
| **Phase 2** | Design Read + 内容关系映射 + 设计词汇表 | 中 | 中 | Phase 1 验证有效 |
| **Phase 3** | Dial 机制（方案 B：约束规划不约束渲染） | 中 | 中 | Phase 2 验证有效 |
| **Phase 4** | Dial→Renderer 执行链（方案 A：代码改动） | 中 | 高 | Phase 3 验证 Dial 声明有效 |
| **Phase 5** | 动效纪律 + 高端配色轮换 | 低 | 低 | Phase 4 完成 |

**Phase 1 是最小可行验证点** — 风险最低、收益最高、不需要任何代码改动。验证 AI Tells 和一致性锁是否真的让 LLM 产出更高质量的 PPT，再决定是否推进后续阶段。
